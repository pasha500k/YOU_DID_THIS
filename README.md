# Telegram + VK RAG Memory Bot

Система состоит из Telegram-бота, VK-бота, локальной RAG-памяти, отдельной админки и отдельного пользовательского сайта. Все они работают поверх одной и той же базы данных и одних и тех же application/service-слоев, поэтому материалы, смены, лимиты, промокоды, заявки, департаменты и настройки синхронизированы между ботами и сайтами.

## Архитектура

- `domain/` - доменные модели и порты
- `application/` - бизнес-логика и use cases
- `application/telegram/` - Telegram-специфичный application-слой
- `application/vk/` - VK-специфичный application-слой
- `infrastructure/telegram/` - aiogram-бот и Telegram-адаптеры
- `infrastructure/vk/` - VK-бот на `vk_api`
- `infrastructure/http/` - отдельная админка и отдельный пользовательский веб-сайт
- `services/` - SQLite, RAG, OpenAI, медиа-обработка

## Что умеет система

- Telegram-бот для пользователей
- VK-бот для пользователей
- отдельный пользовательский сайт на другом порту
- отдельная админка на своем порту
- обязательный выбор департамента при первом использовании
- ежедневные спец-режимы департаментов
- промокоды и бонусные запросы
- заявки на дополнительный доступ
- смены с диапазонами дат
- индексирование текста, фото, аудио, видео и документов
- текстовые ответы из RAG-памяти
- персональные API key, prompt и prompt profile
- подробная статистика пользователей и бан-система

Пользовательский сайт:

## Как синхронизируются сайт и боты

- Telegram-бот, VK-бот, админка и пользовательский сайт используют одну SQLite-базу
- индекс материалов общий
- смены общие
- промокоды общие
- заявки на доступ общие
- департаменты и их спец-лимиты общие
- пользовательские настройки API/prompt общие
- лимиты и бонусные запросы общие

Важно:

- для синхронизации данных конкретного человека админ создает отдельный сайт-аккаунт и привязывает его к нужному Telegram- или VK-`user id`
- статистика и лимиты Telegram/VK разделены по платформам, но внутри одной платформы сайт и бот полностью синхронизированы

## Материалы и смены

- материал с датой автоматически попадает в подходящую смену по диапазону дат
- если материал создается через админку, можно загрузить файл прямо на сайте для AI-анализа
- после этого следующее сообщение администратора в Telegram-группе хранения может использоваться как оригинальный источник материала
- ответы пользователям сейчас текстовые: бот анализирует медиа, запоминает извлеченную информацию, но файлы пользователям не отправляет

## Запуск

1. Установить Python 3.11+
2. Установить `ffmpeg` и `ffprobe`
3. Создать виртуальное окружение
4. Установить зависимости:

```bash
pip install -e .
```

5. Заполнить `.env`
6. Запустить:

```bash
python -m telegram_rag_memory_bot
```

## install.sh

Для Linux-сервера есть отдельный [install.sh](install.sh). Он автоматически:

- ставит системные зависимости
- переносит проект в серверную директорию
- создает `.venv`
- ставит Python-зависимости
- подготавливает `.env`
- создает `systemd`-сервис
- запускает ботов, админку и пользовательский сайт

Базовый вариант:

```bash
sudo bash install.sh
```

Во время запуска `install.sh` сам спрашивает основные значения для `.env`:

- `OPENAI_API_KEY`
- `TELEGRAM_BOT_TOKEN`
- `STORAGE_CHAT_ID`
- `API_VK` и `VK_GROUP_ID`, если нужен VK
- пароль админки и admin user id

Если значение уже сохранено, можно просто нажать `Enter` и оставить его как есть.

При повторном запуске `install.sh` проверяет, была ли установка раньше. Если проект уже установлен и обязательные настройки в `.env` заполнены, скрипт:

- не запускает мастер настройки повторно
- обновляет код проекта
- обновляет Python-зависимости в `.venv`
- заново записывает актуальный `systemd` unit
- перезапускает сервис уже с новой версией

Если нужно принудительно снова открыть мастер `.env`, можно запустить так:

```bash
sudo FORCE_ENV_WIZARD=1 bash install.sh
```

Если нужно, чтобы сервер сам клонировал репозиторий:

```bash
sudo APP_REPO_URL=https://example.com/your-repo.git APP_BRANCH=main bash install.sh
```

По умолчанию скрипт поднимает:

- админку на `8787`
- пользовательский сайт на `8790`
- systemd-сервис `telegram-rag-memory-bot`

После первого запуска проверь и заполни рабочими токенами серверный `.env` в каталоге приложения.

## Mobile App

В проекте есть отдельное мобильное приложение в [mobile_app](mobile_app) на `Expo`, которое работает как кроссплатформенный клиент для `iOS` и `Android`.

Базовый запуск:

```bash
cd mobile_app
npm install
npm run start
```

Дальше можно открыть:

- `a` для Android
- `i` для iOS
- или отсканировать QR-код через Expo Go

Важно:

- приложение подключается к публичному сайту, поэтому использует тот же сайт-логин и ту же общую RAG-память
- приложение открывает фиксированный адрес `https://letovoai.ru/`
- внутри мобильного приложения нет выбора адреса: оно показывает только сам сайт

Для более «чистой» релизной сборки и меньшего числа предупреждений безопасности лучше собирать не preview, а production-профили:

```bash
cd mobile_app
eas build --platform android --profile production-apk
eas build --platform ios --profile production-ios
```

Замечание:

- полностью гарантировать, что антивирус или Play Protect никогда не покажет предупреждение, нельзя
- меньше всего ложных срабатываний обычно у подписанных production-сборок
- для iPhone и более строгих проверок лучше использовать полноценный `https`-адрес сервера, сейчас это `https://letovoai.ru/`

## Правила старта runtime

- Telegram запускается, если заполнен `TELEGRAM_BOT_TOKEN` или `SETTINGS_BOT_TOKEN`
- VK запускается, если заполнен `API_VK`
- админка запускается, если `LOCAL_UPLOAD_ENABLED=true`
- пользовательский сайт запускается, если `PUBLIC_WEB_ENABLED=true`

`VK_BOT_ENABLED` оставлен только для обратной совместимости и больше не управляет фактическим запуском VK-бота.

## Важные переменные окружения

- `OPENAI_API_KEY` - ключ OpenAI
- `TELEGRAM_BOT_TOKEN` - основной Telegram bot token
- `SETTINGS_BOT_TOKEN` - fallback token, если `TELEGRAM_BOT_TOKEN` пуст
- `API_VK` - токен VK-бота
- `VK_GROUP_ID` - id сообщества VK
- `STORAGE_CHAT_ID` - Telegram-группа хранения материалов
- `DATABASE_PATH` - путь к SQLite
- `MEDIA_CACHE_DIR` - кэш временных файлов
- `VIDEO_DOWNLOAD_DIR` - папка локальных загрузок видео
- `LOCAL_UPLOAD_HOST` / `LOCAL_UPLOAD_PORT` - адрес и порт админки
- `LOCAL_UPLOAD_PASSWORD` - пароль админки
- `PUBLIC_WEB_HOST` / `PUBLIC_WEB_PORT` - адрес и порт пользовательского сайта
- сайт использует отдельные логин/пароль, которые админ создает в локальной админке и хранит в таблице `site_accounts`
- `PUBLIC_WEB_PASSWORD` и `BOT_ACCESS_PASSWORD` можно считать legacy-настройками: основной вход пользовательского сайта теперь идет через БД-сайт-аккаунты
- `AUTHORIZED_USER_IDS` - разрешенные Telegram user id при `PUBLIC_ACCESS=false`
- `UPLOADER_USER_IDS` - Telegram-админы
- `VK_AUTHORIZED_USER_IDS` - разрешенные VK user id при `PUBLIC_ACCESS=false`
- `VK_UPLOADER_USER_IDS` - VK-админы
- `DAILY_MESSAGE_LIMIT` - дневной лимит обычных запросов

## Заметки

- личные API key и prompts хранятся локально в SQLite
- длинное аудио режется по длительности через `ffmpeg`
- сайт и боты используют один и тот же application/service-слой, поэтому изменения через сайт сразу видны ботам и наоборот
- если Telegram должен читать обычные сообщения в группах, может понадобиться выключить privacy mode в BotFather
- для VK нужно включить Long Poll API и событие `message_new`
